# Cahier des charges fonctionnel — Élaboration d’un portfolio Data

## 1. Présentation du projet

Ce projet consiste à concevoir un **portfolio professionnel en ligne** permettant de présenter mon profil de Data Analyst / BI & Analytics Engineer dans le cadre d’une candidature pour une mission chez **Aéroworld**, entreprise française internationale du secteur aéronautique.

Le portfolio doit permettre au client d’évaluer rapidement :

- mon profil professionnel ;
- mes projets data réalisés ;
- mes compétences techniques, métier et soft skills ;
- mon adéquation avec les besoins exprimés par Aéroworld ;
- ma capacité à structurer un projet data ;
- ma posture de conseil ;
- ma capacité à documenter, vulgariser et accompagner les utilisateurs.

Le livrable final attendu est un **portfolio en ligne hébergé sur GitHub**, accompagné d’un document contenant le lien vers ce portfolio et regroupant l’ensemble des éléments demandés.

---

## 2. Enjeux et objectifs

## 2.1 Besoin de l’entreprise cliente

Aéroworld recherche un profil de **Data Analyst chef de projet expérimenté et polyvalent**, capable d’évoluer dans un environnement data complexe, international et industriel.

L’entreprise fait face à des enjeux data importants :

- gestion de données massives ;
- collecte et stockage sécurisé des données ;
- interopérabilité entre sources et systèmes ;
- harmonisation des données issues de plusieurs départements ;
- valorisation stratégique des données ;
- aide à la décision ;
- innovation data, machine learning et intelligence artificielle ;
- documentation fiable ;
- accompagnement des équipes ;
- posture de conseil.

Le portfolio doit donc démontrer que mon profil répond à ces attentes à travers des preuves concrètes issues de mes projets.

---

## 2.2 Raisons du besoin

Le client ne souhaite pas seulement consulter un CV. Il souhaite analyser un **portfolio réflexif et visuel** capable de prouver les compétences attendues.

Le besoin porte donc sur un support qui permette de :

- structurer mon profil de manière professionnelle ;
- démontrer les compétences acquises par des projets concrets ;
- relier chaque projet aux attentes du client ;
- présenter des indicateurs pertinents ;
- rendre l’information lisible pour un public métier, RH et data ;
- montrer ma capacité à communiquer clairement ;
- incarner une posture de consultant data.

## 4. Spécifications ergonomiques

## 4.1 Principes ergonomiques du portfolio

Le portfolio doit être conçu comme un support professionnel, lisible et orienté décision. Il doit permettre une consultation rapide tout en donnant accès à des preuves détaillées.

Les principes ergonomiques retenus sont les suivants :

- navigation simple et hiérarchisée ;
- structure claire par sections ;
- titres explicites ;
- textes courts en introduction de chaque section ;
- accès direct aux livrables ;
- cohérence visuelle entre les pages ;
- mise en avant des indicateurs clés ;
- lecture possible par un public non technique ;
- preuve systématique des compétences par projets et livrables.

---

## 4.2 Principes de navigation et UX design

Le portfolio est structuré autour d’un **README principal** servant de page d’accueil et de navigation.

Les sections principales sont :

- Profil expert ;
- Analyse du besoin client ;
- Cahier des charges fonctionnel ;
- Diagramme de Gantt ;
- Dashboards Power BI ;
- Documentation utilisateur ;
- Vidéo de formation ;
- Projets du portfolio ;
- Certifications et CV.

Chaque section dispose de son propre `README.md`, permettant une navigation fluide dans GitHub.

L’utilisateur doit pouvoir comprendre en moins de quelques minutes :

1. qui je suis ;
2. ce que le client attend ;
3. quels projets démontrent mes compétences ;
4. quels indicateurs prouvent l’adéquation de mon profil ;
5. comment utiliser les dashboards et le portfolio.

## 6.2 Technologies utilisées

Le projet portfolio repose sur des outils simples, accessibles et adaptés à un contexte de candidature professionnelle. Les choix techniques privilégient la lisibilité, la facilité de navigation, la maintenabilité et la possibilité de présenter rapidement les livrables au client.

| Outil / technologie | Usage dans le projet |
|---|---|
| **GitHub** | Hébergement du portfolio en ligne, organisation des sections et centralisation des livrables. |
| **Markdown** | Rédaction des pages `README.md` pour structurer chaque section du portfolio. |
| **Power BI** | Création des deux dashboards : profil expert et alignement avec les besoins Aéroworld. |

| **PowerPoint / PDF** | Mise en forme éventuelle des livrables organisationnels : cahier des charges, Gantt, documentation. |
| **Outil de capture vidéo** | Enregistrement de la vidéo de formation et de prise en main du portfolio. |
| **Images PNG / captures d’écran** | Intégration visuelle des dashboards, certificats, CV et livrables dans le portfolio. |

Ces technologies permettent de produire un portfolio à la fois **documenté, visuel, démonstratif et facilement consultable** par un recruteur, un manager data ou un interlocuteur métier.

## 6.3 Données utilisées pour les dashboards

Les dashboards du portfolio sont construits à partir de cinq tables principales. Ces tables ont été créées spécifiquement pour répondre au besoin du client Aéroworld et pour démontrer la pertinence de mon profil à travers des indicateurs mesurables.

| Table | Rôle dans le dashboard |
|---|---|
| **Projects** | Référentiel des projets présentés dans le portfolio : nom, type, domaine, outil technique, outil métier, use case principal et pertinence pour Aéroworld. |
| **ClientUseCases** | Liste des cas d’usage data identifiés dans le besoin Aéroworld : gestion de données massives, Data Lake, interopérabilité, sécurité, maintenance prédictive, innovation, documentation, posture consultant. |
| **ClientRequirements** | Référentiel des compétences attendues par le client : compétences techniques, métier et soft skills, avec un niveau d’importance et un rattachement à un use case. |
| **ProjectSkills** | Liste des compétences démontrées par chaque projet du portfolio, avec un niveau de maîtrise et une preuve associée. |
| **RequirementEvidence** | Table de correspondance entre les attentes client et les projets du portfolio, avec une description de la preuve et un niveau de couverture. |

Ces tables permettent de passer d’un portfolio descriptif à un portfolio analytique, capable de répondre à deux questions clés :

1. **Quelles compétences sont démontrées par mes projets ?**
2. **Dans quelle mesure ces compétences répondent-elles aux attentes d’Aéroworld ?**

## 7.3 Sécurité et sauvegarde

La sécurité du projet repose sur une logique simple : publier uniquement les éléments nécessaires à la démonstration professionnelle du profil, tout en évitant l’exposition de données sensibles ou confidentielles.

### Pratiques de sécurité retenues

- vérification des fichiers avant publication ;
- contrôle des captures d’écran intégrées au portfolio ;
- suppression des informations personnelles inutiles ;
- absence de données confidentielles issues d’un contexte professionnel réel ;
- anonymisation ou retrait des données sensibles lorsque nécessaire ;
- organisation claire des dossiers pour éviter les erreurs de publication ;
- utilisation d’un dépôt GitHub unique pour centraliser les livrables.

### Sauvegarde des livrables

Les livrables sont sauvegardés à deux niveaux :

| Niveau de sauvegarde | Description |
|---|---|
| GitHub | Le dépôt conserve l’historique des modifications et centralise les versions publiées. |
| Sauvegarde locale | Les fichiers sources sont conservés localement : dashboards, documents, images, tableaux et vidéo. |

Cette organisation permet de sécuriser le projet, de suivre les modifications et de limiter le risque de perte de fichiers avant la soutenance.
